Apollo
Completed June 2017

Project Overview

Apollo was designed to be a simple replacement for the current rideshare platforms on facebook. It allowed users to create trips and post them to be found by other users. Users can search for trips by using from and to cities, as well as a date. This project is powered by Ruby On Rails with a MongoDB backend. The frontend is powered by Angular 2.

Functionality

Viewing a trip detail from the driver's perspective.

When a rider visits the application, they are able to perform Login/Signup actions right on the main page, along with the ability to search for trips using search criteria such as from, destination, and time. Once the use finds a trip they would like to take, they can request it after logging in with their own account. After requesting, the driver will then be able to see the request and choose to accept it or not. Once the driver has accepted their passengers, both parties can reach out to each other using social media, phone, or email which will all be provided to both parties after a trip is finalized.

A driver who wishes to post a listing can login to their account and click the create trip option in the top nav bar, where they will be prompted to enter their car details and then enter in the details of their trip. Drivers will be rated out of 5 for their trips and this rating will be visible to anyone who finds one of their trips.

Authentication

Authentication in Apollo was handled with tokens. Every time the user needed to make a request to the rails server, a JWT token would be passed along with the headers of the http request. The JWT Token is initially provided to the web client when the user logs in for the first time. It is then subsequently stored in the browser as a cookie. This cookie would then be deleted upon logging out of the application.

Rails Server

Demonstration of the search trips functionality.

The rails server was capable of handling a wide range of http requests to help add and query data in the database. The rails server offered complete CRUD operations for the user class, as well as being fully able to send email confirmation, forgot password, and update password emails from the application's own email address.

The http endpoints to perform CRUD and query operations on the Trips stored in the database were even more extensive, offering the ability to check listings, check authorization for certain listings, getting the user's trips, checking if a trip was expired, full management of trip requests, and most importantly, full search functionality for trips.

MongoDB Database

All of the data stored for the application was kept on a mongodb database. MongoDB was chosen because it offered a free and easy way to store the application's data. It was also a NoSQL database which was something that I was particularly interested in at the time because of it's flexibility and ease of use. It was also easily compatible with the Ruby on Rails platform.

Hosting

The Ruby on Rails server was hosted on AWS's Elastic Beanstalk environment. This provided an easy way to deploy and monitor the application but eventually the cost of keeping this application running was the reason it was eventually terminated. However, the MongoDB was hosted on a free sandbox node running on MongoLAB, which offered free hosting for any node that was less than 500MB.
